## 4. 待改善項目 — 三個方向

### 4.1 改善方向一：實際測試數據驗證

**問題描述：**
文件中的效能數據（API 回應時間、Token 消耗、錯誤率等）多為基於架構設計的估算值，而非基於實際壓力測試的真實數據。這可能導致決策者對系統效能有不準確的預期。

**具體影響：**
- 白皮書中的效能比較表需要真實數據驗證
- 自我複製計畫中的成本估算需要實際 API 使用量校準
- 安全審計中的某些弱點嚴重性需要滲透測試確認

**改善方案：**
- 建立自動化效能測試套件（使用 k6 或 Artillery）
- 定期執行壓力測試並記錄真實數據
- 將真實數據回填至文件中
- 設定效能基線，後續迭代時進行對比
- 委託第三方進行滲透測試

**預期時間：** 2-3 週

### 4.2 改善方向二：互動式內容與多媒體

**問題描述：**
所有文件皆為靜態的 Markdown 格式。雖然 Markdown 易於版本控制和維護，但對於教學逐字稿和比較分析等文件，缺乏互動性會降低閱讀體驗和理解效率。

**具體影響：**
- 教學逐字稿沒有配套的操作截圖或影片
- 比較分析中的表格無法動態篩選和排序
- 架構圖僅以文字描述，缺乏視覺化圖表

**改善方案：**
- 將教學逐字稿轉化為影片腳本，錄製螢幕操作教學影片
- 使用 Mermaid 或類似工具在 Markdown 中嵌入架構圖
- 建立 Framer 上的互動式文件中心
- 考慮使用 Mintlify 或 Docusaurus 建立文件網站
- 為比較分析添加互動式圖表

**預期時間：** 4-6 週

### 4.3 改善方向三：國際化與多語言支援

**問題描述：**
所有文件目前僅有繁體中文版本。如果 SEOBAIKE 要拓展國際市場（日本、韓國、東南亞、歐美），需要多語言文件版本。

**具體影響：**
- 無法向國際投資者展示產品
- 國際開發者無法閱讀 API 文件
- 限制了潛在合作夥伴的範圍

**改善方案：**
- 建立文件國際化框架（i18n 目錄結構）
- 優先翻譯白皮書和 API 文件為英文
- 次優先翻譯為日文（日本市場進入策略）
- 使用 AI 輔助翻譯 + 人工審校的工作流
- 建立術語對照表（中英日），確保翻譯一致性

**預期時間：** 6-8 週

## 5. 如果重來一次

### 5.1 我會改變的五件事

**第一件：先做安全審計，再做功能開發**

回顧整個開發歷程，安全審計（第 9 層）揭示了多個嚴重弱點，其中最關鍵的是存取權杖硬編碼於版本控制文件中。如果重來，我會在第一個 migration 之前就建立安全基線：設定環境變數管理、啟用 RLS 預設策略、建立審計日誌機制。安全不應該是事後補救，而應該是從第一天就融入架構的設計原則。

**第二件：建立文件優先（Documentation-First）的工作流**

目前的文件是在大量開發工作完成之後才一次性撰寫的。這導致兩個問題：一是文件可能遺漏某些開發過程中的重要決策脈絡；二是需要回憶和重建資訊，效率較低。如果重來，我會在每個功能開發前先寫文件——定義 API 規格、確認資料結構——然後再寫程式碼。

**第三件：從第一天就設計多引擎架構**

從競爭分析和防禦策略中可以清楚看出，沒有任何一個 AI 引擎在所有維度都是最佳選擇。如果重來，我會從架構設計的第一天就規劃 AI 引擎抽象層，讓切換和混用不同引擎變得簡單。這樣可以更早地實現成本最佳化（低價值任務用低成本引擎）和功能互補（不同任務用最適合的引擎）。

**第四件：更早建立自動化測試**

目前的測試覆蓋率不足，功能正確性主要依賴人工驗證。workers 目錄下雖然有一些測試檔案（auth.test.ts、normalizer.test.ts 等），但尚未形成完整的測試體系。如果重來，我會在第一個 Worker 部署之前就建立完整的測試框架。

**第五件：嚴格遵循 CaaS 架構的「等待核准」原則**

在高速開發的壓力下，有時會忍不住跳過「等老闆回覆同意再執行」的步驟。這違反了 CLAUDE.md 中的最高原則。如果重來，我會設計一個技術機制（例如 pre-commit hook）來強制執行核准流程，而不是依賴自覺。

### 5.2 我會保持不變的三件事

**第一件：CaaS 架構（人決策，AI 執行）**

這個核心理念是正確的。在 AI 能力越來越強的時代，「誰來做決策」這個問題變得至關重要。SEOBAIKE 的答案很清楚：人（許竣翔）做決策，AI 做執行。這不僅是技術架構，更是商業倫理和企業治理的基本原則。

**第二件：專利約束層 L1-L4**

四層推理約束機制是 SEOBAIKE 最獨特的差異化優勢。在所有九份文件中，L1-L4 約束層反覆出現在各種脈絡中——白皮書的架構核心、教學中的基礎概念、安全審計中的保護對象、防禦策略中的護城河、未來路線圖中的標準化目標。這證明了它的設計是貫穿整個平台的核心基因。

**第三件：選擇 Supabase + Cloudflare 的技術棧**

這個技術組合在成本效益、開發效率和可擴展性之間取得了很好的平衡。Supabase 提供了 PostgreSQL 的強大功能加上 RLS、Realtime、Edge Functions 等現代化能力；Cloudflare 提供了全球邊緣網路和 Workers 平台。兩者結合，讓一個小團隊也能建構全球化的平台。
